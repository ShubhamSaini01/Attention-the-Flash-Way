Attention the Flash Way

Building a Flash Attention CUDA kernel from scratch — what worked, what failed, and why.

Five kernel iterations. Three bugs found and diagnosed. One working Flash Attention kernel that beats a naive baseline at every sequence length and scales to N=16K+ on 12GB VRAM.

Every performance claim in this presentation is backed by benchmark data or cuobjdump output. CUDA specifications are cited to NVIDIA documentation.

Stage 1 — Naive Attention BASELINE

Three separate kernels, chained sequentially:

Architecture

  1. qk_kernel — S = QKT / √D, one (row, col) per thread
  2. softmax_kernel — serial per-row softmax
  3. sv_kernel — O = S × V

Materializes the full N×N score matrix in global memory.

Why it's fast at small N

  • Embarrassingly parallel — zero sync overhead
  • At N=512, score matrix = 8 MB (H=8) — fits in 48 MB L2
  • Simple memory access patterns → good coalescing
// qk_kernel: N threads per block, one dot product each
dim3 grid_qk(N, H, B);
dim3 block_qk(N);  // ← launches N threads per block
                   //   CUDA max = 1024 → kernel fails silently for N > 1024
Hard limit: The naive kernel launches N threads per block. CUDA caps blocks at 1,024 threads, so this kernel cannot run for sequence length N > 1024.
Source: CUDA C++ Programming Guide §H — "Maximum number of threads per block: 1024"

Stage 2 — Optimized FAILED — 15× SLOWER

Intended to improve naive with shared memory for K and parallel reduction for softmax. Actually 15.2× slower than naive.

20.45ms Median @ N=512 (vs 1.34ms naive)
255/256 Threads wasted per dot product
1,024 __syncthreads() calls per block

The Bug

// qk_kernel_opt: BLOCK_SIZE=256 threads
for (int col = 0; col < N; col++) {
    // All 256 threads cooperatively load K into shared memory
    for (int d = threadIdx.x; d < D; d += blockDim.x)
        K_shared[d] = K[k_idx];
    __syncthreads();

    // But then ALL 256 threads compute the SAME dot product...
    float sum = 0.0f;
    for (int d = 0; d < D; d++)
        sum += Q[q_idx] * K_shared[d];  // ← 255 threads do identical work

    if (threadIdx.x == 0)  // ← only thread 0 writes
        S[s_idx] = sum / sqrtf((float)D);
    __syncthreads();
}
Root cause: All 256 threads independently compute the same dot product, but only thread 0 writes the result. 255 out of 256 threads do completely wasted arithmetic. The outer for col loop serializes N=512 columns with 2×N = 1,024 __syncthreads() barriers.

Benchmark: 20.447ms median vs 1.344ms naive = 15.2× slower. Measured on RTX 5070, 100 runs.

Stage 3 — Flash Attention v1 FAILED — 75× SLOWER

Correct Flash Attention algorithm (online softmax, tiled K/V, register accumulator) but catastrophically wrong thread-to-work mapping.

104ms Median @ N=512 (vs 1.34ms naive)
384B Stack spill / thread (cuobjdump)
4× Wasted thread compute

Bug 1: Stack Spill from Oversized Per-Thread Arrays

// BLOCK_SIZE = 256 threads
float O_local[64];     // 64 floats = 256 bytes per thread
float scores[TILE_SIZE]; // 32 floats = 128 bytes per thread
// Total: 96 floats of per-thread arrays
// Compiler cannot fit these in registers → spills to stack (local memory)
Evidence from cuobjdump:

flash_attention_kernel: REG:31 STACK:384 SHARED:1024
flash_attention_v2_kernel: REG:32 STACK:0 SHARED:1024

The 384 bytes of stack spill per thread means every array access goes through L2 cache → DRAM instead of the register file. With 256 threads per block, that’s 98,304 bytes of local memory traffic per block.

Source: NVIDIA Technical Blog — "How to Improve CUDA Kernel Performance with Shared Memory Register Spilling": "all register spills were placed in local memory, which resides off-chip in device global memory."

Bug 2: All 256 Threads Compute Identical Results

// ALL 256 threads compute the same dot product for each (row, col) pair
for (int d = 0; d < D; d++)
    dot += Q_tile[d] * K_tile[col * D + d];  // ← all threads do this

// Only threads 0..63 write output (D=64, stride=BLOCK_SIZE=256)
for (int d = tid; d < D; d += blockDim.x)  // ← threads 64-255 write nothing
    O[o_idx] = O_local[d] / sum_val;

Root Cause Analysis

Both failed kernels share the same fundamental mistake: threads are not mapped to independent output dimensions.

Broken pattern

256 threads → each thread computes the full dot product (all D=64 dimensions) → only 1 or 64 threads write output.

  • Wasted compute: 4× to 256×
  • Oversized per-thread arrays
  • Stack spill to local memory
  • Excessive sync barriers

Correct pattern

32 threads → each thread owns D/32 = 2 dimensions → cooperative dot product via warp shuffle → every thread writes its own output.

  • Zero wasted compute
  • Scalar accumulators (2 floats/thread)
  • No stack spill
  • __syncwarp() only (no block sync)

The key insight: warp shuffle as broadcast reduction

// Butterfly reduction via __shfl_xor_sync
// After 5 rounds, ALL 32 threads hold the total sum.
// Unlike __shfl_down (where only thread 0 gets the result),
// __shfl_xor is symmetric: thread i exchanges with thread i^offset.
__device__ __forceinline__ float warp_reduce_sum(float val) {
    for (int offset = 16; offset > 0; offset >>= 1)
        val += __shfl_xor_sync(0xFFFFFFFF, val, offset);
    return val;  // ← every thread has the full sum
}
Why this works: __shfl_xor_sync(mask, val, offset) returns the value from lane lane_id ^ offset. Because XOR is symmetric (thread i reads from i^offset, and i^offset reads from i), after 5 rounds (offsets 16, 8, 4, 2, 1) all 32 threads converge to the same total sum. No shared memory needed.

Source: NVIDIA Developer Blog — "Using CUDA Warp-Level Primitives" (2018); CUDA C++ Programming Guide §B.15 — Warp Shuffle Functions.

One warp (32 threads) per query row. Each thread owns D/32 = 2 output dimensions for the entire kernel lifetime.

Online softmax — per-element, no scores array

for (int col = 0; col < tile_size; col++) {
    float score = warp_reduce_sum(partial) / sqrtf((float)D);

    float new_max = fmaxf(row_max, score);
    float exp_old = expf(row_max - new_max);
    float exp_new = expf(score  - new_max);

    o_local[i] *= exp_old;              // rescale running accumulator
    row_sum = row_sum * exp_old + exp_new;
    o_local[i] += exp_new * V_s[...];   // accumulate weighted V
    row_max = new_max;
}
Why 2.58ms > naive 1.34ms: v2 uses only 32 threads per block. The GPU has 48 SMs × 2048 max threads/SM = 98,304 thread capacity, but v2 only launches 512×8×1 = 4,096 blocks × 32 = 131,072 threads. Low occupancy per SM (only 1 active warp per block) limits instruction-level parallelism and latency hiding.

Source: CUDA C++ Programming Guide §5.2 — "Maximize Utilization": occupancy is the ratio of active warps to maximum warps per SM.

Stage 5 — Flash v3 (Multi-Row) FASTEST CUSTOM KERNEL

Packs 4 warps into one block, each handling a different query row. The critical insight: all rows in the same (batch, head) share the same K and V.

0.77ms Median @ N=512
1.75× Faster than naive
3.4× Faster than v2

K/V tile sharing across warps

// Grid: (N/4, H, B)  Block: 128 threads = 4 warps
// Shared memory layout:
//   Q_s: 4 × D floats  (each warp has its own Q row)
//   K_s: TILE × D floats (SHARED across all 4 warps) ← key insight
//   V_s: TILE × D floats (SHARED across all 4 warps)

// ALL 128 threads cooperatively load K/V (coalesced)
for (int i = tid; i < tile_size * D; i += 128) {
    K_s[i] = K[kv_base + i];
    V_s[i] = V[kv_base + i];
}
__syncthreads();  // full block sync — all warps need K_s, V_s

// Each warp independently processes its own query row
// (warp shuffle for dot products, no cross-warp sync needed)

Why v3 is faster: three compounding benefits

1. Amortized K/V bandwidth

K/V tiles loaded once, reused by 4 warps = 4× less global memory traffic for K/V reads.

2. Higher occupancy

128 threads/block vs 32 = 4× more active threads per SM, better latency hiding.

3. Faster tile loads

128 threads cooperating on coalesced loads vs 32 = 4× higher load throughput. Each thread loads tile_size×D/128 = 16 elements instead of 64.

Shared memory budget

Per-block shared memory: (4×64 + 2×32×64) × 4 = 17,408 bytes (17 KB)
RTX 5070 provides 100 KB shared memory per SM → up to 5 blocks per SM from shared memory alone.
Register limit: 40 regs/thread × 128 threads = 5,120 regs/block. 65,536 per SM → 12 blocks. Shared memory is the binding constraint at 5 blocks/SM.

Source: CUDA C++ Programming Guide §H — Compute Capabilities: 65,536 registers per SM; cuobjdump: Flash v3 = 40 REG.

Key Takeaways

1. Map threads to output dimensions, not input iterations

The optimized and v1 kernels failed because many threads computed the same result. The fix: each thread owns a unique subset of the output and cooperates on inputs via warp shuffle. This is the single most important design decision in the kernel.

2. Stack spill matters more than register count

v1 and v2 use nearly identical register counts (31 vs 32). The performance difference is the 384-byte stack spill in v1 — invisible without cuobjdump. Per-thread arrays that exceed the register file are silently placed in local memory, turning register-speed accesses into DRAM-speed accesses.

3. Share K/V tiles across query rows

All query rows in the same (batch, head) read the same K and V. Packing multiple rows per block amortizes the global memory load cost. This gave a 3.4× speedup (v2 → v3) with no algorithmic change.

4. Flash Attention is a memory optimization, not a compute optimization

At N=512, the N×N matrix fits in L2 and naive is simple. Flash wins at scale because it never allocates O(N²) memory. At N=16K, naive would need 8 GB for scores alone; Flash uses 32 MB total. The algorithm trades more compute (online softmax rescaling) for radically less memory.

5. Profile before and after — gut instinct fails

The “optimized” kernel was 15× slower. Flash v1 was 75× slower. Both were written with reasonable intuitions (shared memory is fast, fused kernels are fast). Without benchmarks and cuobjdump, these regressions would have shipped as “optimizations.”
